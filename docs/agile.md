# Investigación de Metodologías de Desarrollo (Agile)

## 1. ¿Qué es Agile y cuál es su objetivo?
Agile (o desarrollo ágil) no es una herramienta concreta, sino una filosofía y un conjunto de principios para la gestión de proyectos de software. Su objetivo principal es entregar software funcional de valor de forma rápida, continua y adaptativa. 

En lugar de planificar todo el proyecto de forma rígida durante meses (enfoque tradicional en cascada), Agile divide el desarrollo en ciclos pequeños para responder de manera flexible a los cambios del entorno y a los comentarios reales del usuario.

## 2. Metodología Scrum: Conceptos Clave
Scrum es un marco de trabajo ágil estructurado que organiza el desarrollo en bloques de tiempo fijos y repetitivos. Sus pilares fundamentales son:

* **Roles Clave:**
    * *Product Owner:* El responsable de definir las prioridades del negocio y lo que se debe construir.
    * *Scrum Master:* El facilitador del equipo, encargado de eliminar cualquier obstáculo o bloqueo técnico y asegurar que el proceso fluya.
    * *Developers/Equipo de desarrollo:* Los profesionales técnicos que construyen el software.
* **Sprint:** Es el corazón de Scrum. Es un ciclo de trabajo cerrado que suele durar entre 1 y 4 semanas donde se pasa de una idea a un incremento de software real y totalmente funcional.
* **Backlog:** La lista maestra y priorizada de todas las tareas, características y requisitos pendientes de la aplicación.
* **Reviews (Revisiones):** Reuniones al final de cada Sprint donde el equipo muestra el software terminado a los interesados para recibir críticas constructivas y ajustar la dirección del proyecto.

## 3. Metodología Kanban
Kanban es un sistema visual enfocado en la gestión del flujo de trabajo continuo. Utiliza un tablero dividido en columnas (típicamente: *Por hacer*, *En proceso*, *Hecho*) donde las tareas se representan mediante tarjetas dinámicas que se desplazan de izquierda a derecha. Su meta principal es optimizar la eficiencia y limitar el trabajo en curso (WIP - Work in Progress) para evitar cuellos de botella.

---

## 4. Comparativa Estratégica: Scrum vs. Kanban

| Característica | Scrum | Kanban |
| :--- | :--- | :--- |
| **Estructura** | Muy definido, con roles rígidos y eventos fijos. | Flexible, sin roles obligatorios, flujo continuo. |
| **Bloques de tiempo** | El trabajo se divide en Sprints estricto (1-4 semanas). | No hay fechas límite fijas por ciclos; flujo continuo. |
| **Métricas clave** | Velocidad del Sprint (puntos de historia completados). | Tiempo de ciclo (cuánto tarda una tarea en completarse). |
| **Gestión de cambios** | No se permiten modificaciones dentro de un Sprint activo. | Se pueden añadir o repriorizar tareas en cualquier momento. |

---

## 5. Criterios de Aplicación del Proyecto
* **Cuándo usar Scrum:** Es ideal para proyectos grandes con metas complejas a medio plazo, donde se requiere una sincronización matemática del equipo y revisiones periódicas con el cliente para validar el avance estructural.
* **Cuándo usar Kanban:** Es perfecto para proyectos de mantenimiento continuo, soporte técnico o aplicaciones de desarrollo individualizado (como las *Microapps* propuestas), donde las prioridades cambian diariamente y se necesita visualizar el cuello de botella de manera inmediata sin la burocracia de reuniones rígidas.
